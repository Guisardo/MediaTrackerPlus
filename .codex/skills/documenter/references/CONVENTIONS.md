# MediaTrackerPlus Documentation Conventions

Project-specific patterns observed in and required by the MediaTrackerPlus codebase. These rules extend the generic normalization and organization rules.

---

## Project Identity

| Field | Value |
|-------|-------|
| Project name | **MediaTrackerPlus** (one word, camel case with "Plus") |
| Short form | MTP (only in technical contexts) |
| Repo | `MediaTrackerPlus` |
| Upstream fork | `MediaTracker` by dnlwttnbrg |
| License | MIT |

Use the full name "MediaTrackerPlus" on first mention per document; abbreviation may follow in parentheses if needed.

---

## README Structure Convention

The root `README.md` must follow this section order:

```
1. Badges row
2. H1 project title
3. Short description (1–2 sentences)
4. H2 Features
5. H2 Installation
   H3 From Source
   H3 Docker
   H3 docker-compose
6. H2 Configuration
   H3 Environment Variables (table)
7. H2 Metadata Providers (table)
8. H2 Notifications (table)
9. H2 Integrations
10. H2 Credits / License
```

If a section grows beyond ~30 lines, extract it to `docs/` and replace with a summary + link.

---

## Badge Format

Badges appear on the first line of `README.md`, before the H1. Use shields.io format:

```markdown
[![Build Status](https://github.com/OWNER/MediaTrackerPlus/actions/workflows/WORKFLOW.yml/badge.svg)](https://github.com/OWNER/MediaTrackerPlus/actions)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE.md)
[![Docker Pulls](https://img.shields.io/docker/pulls/OWNER/mediatrackerplus)](https://hub.docker.com/r/OWNER/mediatrackerplus)
```

Rules:
- Badges on a **single line** separated by spaces (not separate lines)
- Alt text must describe the badge, not repeat the URL
- Link each badge to its relevant page

## Environment Variables Table

The canonical format for environment variable documentation:

```markdown
| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `PORT` | No | `9000` | HTTP port the server listens on |
| `DB_PATH` | No | `./data/db.sqlite3` | Path to SQLite database file |
| `TMDB_API_KEY` | Yes* | — | The Movie Database API key |
```

Rules:
- Variable names in backtick inline code
- Default values in backtick inline code, or `—` if none
- "Yes*" for conditionally required (explain in a note below the table)
- Sort by: required first, then optional alphabetically

## Code Example Conventions

For shell commands in README, always show the command in context:

```bash
# Clone and install
git clone https://github.com/OWNER/MediaTrackerPlus.git
cd MediaTrackerPlus
npm install
npm run build
```

For Docker commands:

```bash
docker run -d \
  --name mediatracker \
  -p 7481:7481 \
  -v /path/to/data:/data \
  ghcr.io/OWNER/mediatrackerplus:latest
```

For docker-compose, use the `yaml` language identifier:

```yaml
version: "3"
services:
  mediatracker:
    image: ghcr.io/OWNER/mediatrackerplus:latest
    ports:
      - "7481:7481"
    volumes:
      - ./data:/data
```

## API Documentation Convention

The `server/openapi.json` is the authoritative source for API documentation. Rules:

- Never duplicate endpoint descriptions in Markdown — always generate from the spec
- Reference the hosted docs: `https://dnlwttnbrg.github.io/MediaTrackerPlus/`
- When documenting a specific endpoint in a guide, link to its OpenAPI definition

---

## Monorepo Documentation Layout

Because this is a `server/` + `client/` monorepo:

- Root `README.md` covers the whole system (setup, configuration, deployment)
- `server/README.md` must **not** duplicate root content — it should either:
  - Redirect: `See the [main README](../README.md) for full documentation.`
  - Or contain only server-specific developer notes (testing, DB migrations)
- `client/` has no README currently — if added, scope it to frontend development

---

## Versioning in Docs

Version numbers follow **semver**: `MAJOR.MINOR.PATCH`.

- Current: `v0.1.0` (see `app.json` / latest git tag)
- Reference version in docs as: `v0.1.0` (with `v` prefix)
- `CHANGELOG.md` sections: `## [0.1.0] - YYYY-MM-DD` (no `v` in CHANGELOG headers, per keepachangelog.com)

---

## Internationalization Note

The codebase uses i18next in both `server/` and `client/`. Documentation is written in **English only**. Translation of the UI is separate from documentation.
